# GitLab评论提炼规则 Skill 使用指南

## 简介

这个skill可以从GitLab Merge Request的评论中自动提取代码审查问题，使用AI分析问题背后的逻辑，提炼成通用的代码设计原则和规范文档。

## 核心特性

- **AI语义分析**：使用AI理解评论语义，而非简单的关键词匹配
- **智能提炼**：从具体问题抽象到通用设计原则
- **结构化输出**：生成标准化的规则文档和检查清单
- **无需代码**：只分析评论文本，不依赖代码获取

## 快速开始

### 基本用法

```
从GitLab评论提炼规则，URL: https://gitlab.futunn.com/fcmp/resource_center/-/merge_requests/39
```

### 指定输出文件

```
从GitLab评论提炼规则，URL: https://gitlab.futunn.com/fcmp/resource_center/-/merge_requests/39，输出到: my_rules.md
```

### 只生成通用原则

```
从GitLab评论提炼规则，URL: https://gitlab.futunn.com/fcmp/resource_center/-/merge_requests/39，规则层次: general
```

## 工作流程

### 1. 获取评论数据
- 使用浏览器MCP工具访问GitLab MR页面
- 提取所有评论的文本内容
- 提取元数据（作者、时间、文件、行号）

### 2. 整理评论数据
- 识别问题和解决方案的配对
- 按时间顺序组织
- 去除重复和无用信息

### 3. AI分析评论（核心步骤）
- 使用AI分析每个问答对
- 理解问题背后的核心逻辑
- 提炼解决方案的设计思想
- 生成通用的设计原则

### 4. 分类和归纳
- 将提炼的原则按类别分组
- 合并相似的原则
- 识别设计模式

### 5. 生成规则文档
- 按照标准格式组织规则
- 包含核心思想、具体规则、示例
- 生成检查清单

### 6. 保存文档
- 保存为Markdown文件
- 包含目录和索引

## 输出示例

生成的文档包含：

1. **通用代码设计原则**（按分类组织）
   - 类型系统设计原则
   - 代码组织原则
   - 命名规范原则
   - 数据一致性原则
   - 代码质量原则
   - 等等...

2. **每个原则包含**：
   - 核心思想（一句话概括）
   - 具体规则（3-5条可操作规则）
   - 适用场景
   - 代码示例（正确✅和错误❌对比）

3. **实践检查清单**
   - 按分类组织的检查项
   - 可直接用于代码审查

## 使用场景

- **代码审查后总结**：从代码审查中提炼团队规范
- **建立代码规范**：基于实际审查经验建立规范文档
- **学习最佳实践**：从历史MR中学习设计原则
- **生成检查清单**：为代码审查生成检查清单

## 注意事项

1. **权限要求**：需要GitLab访问权限，可能需要登录
2. **处理时间**：大量评论可能需要较长时间处理
3. **人工审核**：某些规则可能需要人工审核和补充
4. **评论质量**：规则提炼的质量取决于评论的详细程度
5. **AI理解**：AI可能无法完全理解某些复杂的业务逻辑

## 技术实现

### 数据提取
- 使用浏览器MCP工具访问GitLab
- JavaScript提取评论数据
- 只提取文本，不获取代码

### AI分析
- 使用自然语言提示词指导AI分析
- 理解评论语义，提炼设计思想
- 从具体问题抽象到通用原则

### 文档生成
- Markdown格式化
- 结构化的原则组织
- 代码高亮和示例

## 最佳实践

1. **评论质量**：确保代码审查评论详细、准确
2. **问题描述**：问题描述应该清晰，说明为什么有问题
3. **解决方案**：解决方案应该说明设计思想，而不仅仅是技术实现
4. **持续改进**：定期回顾和更新提炼的规则

## 常见问题

### Q: 为什么只提取评论文本，不获取代码？
A: 代码获取不方便，且AI可以通过评论文本理解问题。如果需要代码示例，可以在评论中提供。

### Q: AI分析的结果准确吗？
A: AI分析的结果通常很好，但复杂问题可能需要人工审核。建议结合实际情况使用。

### Q: 可以处理多个MR吗？
A: 当前版本支持单个MR，未来版本可能支持多个MR合并提炼。

### Q: 生成的规则可以修改吗？
A: 可以，生成的文档是Markdown格式，可以直接编辑。
