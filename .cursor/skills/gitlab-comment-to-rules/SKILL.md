# GitLab评论提炼规则 Skill

## When to Use This Skill

此skill在以下场景激活：
- 用户需要从GitLab MR评论中提炼代码规范
- 代码审查后需要总结设计原则
- 建立团队代码规范文档
- 从历史MR中学习最佳实践
- 生成代码审查检查清单

触发关键词：
- "从GitLab评论提炼规则"
- "提炼MR评论成规范"
- "分析代码审查评论"
- "生成代码设计原则"

## Architecture

此skill使用**AI语义分析**方法，而非简单的关键词匹配：

1. **数据提取层**：使用浏览器MCP工具提取GitLab评论
2. **AI分析层**：使用AI理解评论语义，提炼设计思想
3. **规则生成层**：将分析结果组织成结构化规则文档

## Workflow

### Step 1: 获取评论数据

使用浏览器MCP工具访问GitLab MR页面并提取评论：

```javascript
// 1. 导航到GitLab MR页面
browser_navigate(url: gitlab_mr_url)

// 2. 等待页面加载
browser_wait_for(time: 3)

// 3. 提取所有评论
browser_evaluate(function: extract_comments)
```

**提取的数据包括**：
- 评论文本内容
- 作者信息
- 时间戳
- 关联的文件和行号（如果有）
- 评论的层级关系（主题和回复）

**注意**：只提取评论文本，不获取代码片段。代码获取不方便且不是必需的。

### Step 2: 整理评论数据

1. **识别问答对**：
   - 识别问题类型的评论（包含"为什么"、"怎么"、"是否"等疑问词）
   - 识别解决方案类型的评论（包含"done"、"应该"、"需要"等）
   - 按时间和作者匹配问答对

2. **去重和清理**：
   - 去除重复的评论
   - 去除无意义的评论（如"👍"、"LGTM"等）
   - 保留有实质内容的评论

3. **组织数据**：
   - 按时间顺序排列
   - 保持评论的上下文关系

### Step 3: AI分析评论（核心步骤）

对每个问答对，使用AI进行深度分析。**这是整个流程的核心**，使用以下提示词模板：

```markdown
请分析以下GitLab代码审查评论，提炼出通用的代码设计原则：

**问题**：
{问题文本}

**解决方案**：
{解决方案文本}

**上下文信息**：
- 文件：{文件路径}
- 行号：{行号}
- 作者：{作者}

请回答以下问题，并提炼出通用的设计原则：

1. **问题本质**：这个问题的核心是什么？为什么会出现这个问题？
   - 是技术实现问题还是设计思想问题？
   - 反映了什么设计原则的缺失？

2. **解决方案思想**：解决方案背后的设计思想是什么？
   - 遵循了什么设计原则？
   - 体现了什么最佳实践？

3. **通用性分析**：这个原则可以应用到哪些场景？
   - 是否只适用于当前场景，还是可以推广？
   - 类似的问题在哪些地方可能出现？

4. **原则提炼**：提炼出一条通用的设计原则，包括：
   - **原则名称**：简洁明确的名称（如"类型应该准确表达业务语义"）
   - **核心思想**：一句话概括核心思想
   - **具体规则**：3-5条可操作的具体规则
   - **适用场景**：说明在什么情况下应用此原则
   - **代码示例**：提供正确✅和错误❌的代码对比（如果可能）

请用中文回答，格式清晰，便于后续整理成文档。
```

**AI分析的关键点**：
- 理解问题背后的逻辑，而非表面现象
- 从具体问题抽象到通用原则
- 识别设计模式和最佳实践
- 考虑原则的适用性和边界

### Step 4: 分类和归纳

将AI分析的结果进行分类和归纳：

1. **按主题分类**：
   - 类型系统设计
   - 代码组织
   - 命名规范
   - 数据一致性
   - 代码质量
   - 抽象层次
   - 数据模型设计
   - 错误处理
   - 可维护性
   - 其他

2. **合并相似原则**：
   - 识别表达相同思想的不同原则
   - 合并为一条更通用的原则
   - 保留所有相关的具体规则

3. **识别设计模式**：
   - 识别常见的设计模式
   - 总结最佳实践

### Step 5: 生成规则文档

按照以下标准格式组织规则文档：

```markdown
# 通用代码设计原则（从 MR !{number} 提炼）

> 本文档从具体代码审查问题中提炼出通用的设计原则和最佳实践，适用于所有Go项目。

## 一、类型系统设计原则

### 原则 1.1：{原则名称}
**核心思想**：{核心思想描述}

**具体规则**：
- {规则1}
- {规则2}
- {规则3}

**适用场景**：
- {场景1}
- {场景2}

**示例**：
```go
// ✅ 正确
{正确代码示例}

// ❌ 错误
{错误代码示例}
```

## 实践检查清单

### 类型系统设计
- [ ] {检查项1}
- [ ] {检查项2}
```

**文档结构要求**：
- 清晰的目录结构
- 每个原则包含：核心思想、具体规则、适用场景、示例
- 生成可执行的检查清单
- 使用中文编写

### Step 6: 保存文档

将生成的规则文档保存到指定文件：

- 默认文件名：`code_rules_from_mr{number}.md`
- 可自定义输出路径
- 确保文件格式正确（Markdown）

## AI分析提示词模板

### 基础分析模板

```markdown
分析以下代码审查评论，提炼设计原则：

问题：{问题}
解决：{解决方案}

请提炼：
1. 问题本质和原因
2. 解决方案的设计思想
3. 通用原则（名称、核心思想、具体规则、示例）
```

### 深度分析模板（用于复杂问题）

```markdown
深度分析以下代码审查问题：

**原始问题**：
{完整问题文本}

**解决方案**：
{完整解决方案文本}

**代码上下文**（如果有）：
{相关代码片段}

请进行以下分析：

1. **问题分析**：
   - 表面问题是什么？
   - 深层原因是什么？
   - 反映了什么设计缺陷？

2. **解决方案分析**：
   - 解决方案的技术细节
   - 背后的设计思想
   - 遵循的原则和模式

3. **原则提炼**：
   - 原则名称（简洁明确）
   - 核心思想（一句话）
   - 具体规则（3-5条，可操作）
   - 适用场景（何时应用）
   - 代码示例（正确vs错误）

4. **扩展思考**：
   - 类似问题在哪里可能出现？
   - 如何预防此类问题？
   - 相关的设计模式或最佳实践？
```

## Output Format

### 标准输出格式

生成的规则文档必须遵循以下格式：

```markdown
# 通用代码设计原则（从 MR !{number} 提炼）

> 本文档从具体代码审查问题中提炼出通用的设计原则和最佳实践，适用于所有Go项目。

## 目录
- [一、类型系统设计原则](#一类型系统设计原则)
- [二、代码组织原则](#二代码组织原则)
- ...

## 一、类型系统设计原则

### 原则 1.1：类型应该准确表达业务语义
**核心思想**：类型系统不仅是技术实现，更是业务语义的表达工具。

**具体规则**：
- **可空性建模**：字段是否可能为空，应该通过类型系统明确表达
  - 可能为空 → 使用指针类型 `*T`
  - 必须有值 → 使用值类型 `T`
  - **判断依据**：根据业务逻辑和API文档，而非技术便利性

- **类型区分**：语义不同的数据应该使用不同的类型
  - 系统盘和数据盘应该分开定义，即使结构相似
  - 避免使用通用类型承载不同语义的数据
  - **目的**：提高类型安全性，减少误用

**适用场景**：
- 设计数据结构时
- 定义API接口时
- 处理可选字段时

**示例**：
```go
// ✅ 正确：类型明确表达语义
type SystemDisk struct {
    ID   *string `json:"id"`   // 可能为空
    Size int     `json:"size"` // 必须有值
}

type DataDisk struct {
    ID   *string `json:"id"`
    Size int     `json:"size"`
}

// ❌ 错误：类型无法区分语义
type Disk struct {
    Type string  // 通过字段区分，容易混淆
    ID   *string
}
```

## 实践检查清单

### 类型系统设计
- [ ] 字段的可空性是否通过类型系统明确表达？
- [ ] 语义不同的数据是否使用了不同的类型？
- [ ] 是否避免了类型滥用（如用数组存储不同类型数据）？

### 代码组织
- [ ] 相关字段是否分组放置？
- [ ] 是否避免了重复定义？
- [ ] 嵌入结构是否顶格放置？
```

## Technical Notes

### 数据提取注意事项

1. **只提取文本**：
   - 只提取评论文本内容
   - 不尝试获取关联的代码片段
   - 代码获取不方便且不是必需的

2. **处理动态加载**：
   - GitLab评论可能动态加载
   - 需要等待内容完全加载
   - 可能需要滚动页面

3. **处理登录**：
   - 如果页面需要登录，提示用户
   - 等待用户登录后再继续

### AI分析注意事项

1. **提示词质量**：
   - 使用清晰的提示词指导AI分析
   - 明确要求输出格式
   - 要求AI理解问题背后的逻辑

2. **批量处理**：
   - 可以批量分析多个问答对
   - 但要控制每次分析的上下文大小
   - 避免超出AI的上下文限制

3. **结果验证**：
   - AI分析的结果可能需要人工审核
   - 某些复杂问题可能需要多次分析
   - 可以要求AI提供分析依据

### 文档生成注意事项

1. **格式一致性**：
   - 确保所有原则使用相同的格式
   - 保持代码示例的风格一致
   - 使用统一的术语

2. **可读性**：
   - 使用清晰的标题和层级
   - 代码示例要有注释说明
   - 检查清单要具体可操作

3. **完整性**：
   - 确保所有原则都包含必要的信息
   - 检查清单要覆盖所有原则
   - 提供目录便于导航

## Limitations

1. **评论质量依赖**：规则提炼的质量取决于评论的详细程度和准确性
2. **AI理解限制**：AI可能无法完全理解某些复杂的业务逻辑
3. **需要人工审核**：生成的规则可能需要人工审核和补充
4. **上下文缺失**：不获取代码可能导致某些规则缺少具体示例
