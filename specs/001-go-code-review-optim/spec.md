# Feature Specification: Go Code Review Optimization

**Feature Branch**: `001-go-code-review-optim`
**Created**: 2026-01-12
**Status**: Draft
**Input**: User description: "我希望优化go code review,提高代码问题检出率。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Critical Issue Detection Enhancement (Priority: P1)

作为FUTU Go开发者，当我提交代码审查时，我希望系统能够自动检测出所有P0级别的关键问题（错误处理、空指针、GORM操作、并发安全、JSON处理等），以便在代码合并前修复这些必须修复的问题，避免生产环境故障。

**Why this priority**: P0级别问题直接影响系统稳定性和数据安全性，是代码审查的首要目标。目前检出率80%意味着仍有20%的关键问题可能被遗漏，这是不可接受的风险。

**Independent Test**: 可以通过运行代码审查工具检查包含已知P0问题的测试代码文件，验证是否能检出至少95%的关键问题。测试结果独立可衡量，无需其他功能依赖。

**Acceptance Scenarios**:

1. **Given** 包含未使用struct字段的GORM模型代码，**When** 运行代码审查，**Then** 系统应标记出未在Create/Update/Select中使用的字段（规则1.3.7）
2. **Given** 包含循环中调用db.Create()的代码，**When** 运行代码审查，**Then** 系统应建议使用批量操作（规则1.3.10）
3. **Given** 包含使用非ID字段进行Update/Delete的代码，**When** 运行代码审查，**Then** 系统应标记WHERE条件中的非ID定位字段（规则1.3.9）
4. **Given** 包含struct字段顺序不符合规范的代码，**When** 运行代码审查，**Then** 系统应验证ID→业务字段→创建者/修改者→时间戳的顺序（规则1.3.8）
5. **Given** 包含指针解引用但无nil检查的代码，**When** 运行代码审查，**Then** 系统应标记实际的解引用操作而非安全的指针赋值（规则1.2.1优化）

### User Story 2 - Code Quality Issue Detection Improvement (Priority: P2)

作为FUTU Go开发者，当我提交代码审查时，我希望系统能够检测出P1级别的代码质量问题（命名规范、日志标准、代码组织、接口设计等），以便保持团队代码风格一致性和可维护性。

**Why this priority**: P1级别问题影响代码可读性、可维护性和团队协作效率。虽然不会立即导致系统故障，但长期累积会严重降低开发效率。目前40-60%的检出率需要提升到92%以上。

**Independent Test**: 可以通过运行代码审查工具检查包含已知P1问题的测试代码文件，验证命名、日志、组织相关规则的检出率是否达到92%。

**Acceptance Scenarios**:

1. **Given** 包含分层日志记录（内层函数和外层函数都记录同一错误）的代码，**When** 运行代码审查，**Then** 系统应构建调用层次并标记冗余日志（规则2.2.7）
2. **Given** 包含提前return nil但无日志记录的代码，**When** 运行代码审查，**Then** 系统应标记缺少日志的提前返回（规则2.2.8）
3. **Given** 包含通用命名（tmp、getData、process等）的代码，**When** 运行代码审查，**Then** 系统应标记语义不明确的命名并建议具体的替代方案（规则2.1.13-14）
4. **Given** 包含超过50行的函数代码，**When** 运行代码审查，**Then** 系统应计算非注释行数并标记过长函数（规则2.5.3）
5. **Given** 包含context.Context不在首位的函数签名代码，**When** 运行代码审查，**Then** 系统应解析函数签名并验证ctx参数位置（规则2.4.4）
6. **Given** 包含多个相同类型未命名返回值的函数代码，**When** 运行代码审查，**Then** 系统应标记`(int, int, error)`并建议命名返回值（规则2.4.7）

### User Story 3 - Project Structure and Style Consistency (Priority: P3)

作为FUTU Go开发者，当我提交代码审查时，我希望系统能够检测出P2级别的项目结构和代码风格问题（import分组、项目布局、测试标准等），以便遵循最佳实践并保持项目整洁。

**Why this priority**: P2级别问题主要影响代码风格一致性和项目组织规范。虽然优先级较低，但对于大型项目的长期维护至关重要。目前20-40%的检出率需要提升到88%以上。

**Independent Test**: 可以通过运行代码审查工具检查包含已知P2问题的测试代码文件，验证项目结构相关规则的检出率是否达到88%。

**Acceptance Scenarios**:

1. **Given** 包含import语句但未按stdlib→第三方→内部分组的代码，**When** 运行代码审查，**Then** 系统应检查import顺序并验证分组间的空行分隔（规则3.1.2）
2. **Given** 包含项目结构不符合规范的代码，**When** 运行代码审查，**Then** 系统应验证目录组织和文件命名约定（规则3.1.*）
3. **Given** 包含测试覆盖率不足的代码，**When** 运行代码审查，**Then** 系统应检查测试文件存在性和测试用例完整性（规则3.2.*）

### User Story 4 - Comprehensive Test Coverage Validation (Priority: P1)

作为Go代码审查工具维护者，我希望能够通过完整的测试用例覆盖所有97条规则，以便量化验证工具的检出能力并持续改进。

**Why this priority**: 测试覆盖是验证工具能力的基础。当前只有35/97条规则有测试用例（36%覆盖率），无法准确评估工具的实际检出率。这是达成95%检出率目标的必要条件。

**Independent Test**: 可以通过运行新的validation脚本检查所有97条规则的测试用例，验证测试覆盖率是否达到100%，并生成详细的检出率报告。

**Acceptance Scenarios**:

1. **Given** 包含13条GORM规则违规的测试文件（gorm_p0_issues.go），**When** 运行代码审查，**Then** 系统应检出至少13个P0问题并正确标记规则ID
2. **Given** 包含22条命名日志规则违规的测试文件（naming_logging_p1_issues.go），**When** 运行代码审查，**Then** 系统应检出至少20个P1问题（92%）
3. **Given** 包含18条项目结构规则违规的测试文件（structure_p2_issues.go），**When** 运行代码审查，**Then** 系统应检出至少16个P2问题（88%）
4. **Given** 包含0个违规的优秀代码示例文件（all_best_practices.go），**When** 运行代码审查，**Then** 系统应产生≤2%的误报率（假阳性）
5. **Given** 完整的测试套件，**When** 运行validation脚本，**Then** 系统应生成包含每条规则检出率、分优先级统计、综合得分（加权平均）的详细报告

### Edge Cases

- **What happens when** 代码中使用反射或动态字段访问？System应在报告中注明"可能存在动态使用，需人工确认"的免责声明（针对规则1.3.7未使用字段检测）
- **What happens when** 函数中包含大量生成代码或注释？System应在计算函数长度时排除注释行，但注明"不包含生成代码检测"的限制（针对规则2.5.3函数长度）
- **What happens when** 代码中使用闭包或匿名函数进行日志记录？System应在报告中注明"可能无法完整追踪闭包调用链"的限制（针对规则2.2.7分层日志）
- **What happens when** struct字段用于审计目的（CreateTime、UpdateTime等）从不在业务逻辑中读取？System应有明确的例外列表，不标记审计字段为未使用（针对规则1.3.7）
- **What happens when** 批量操作的阈值不明确（多少次循环才算需要批处理）？System应采用合理默认值（如3次以上），并在文档中明确说明阈值设定（针对规则1.3.10批量操作）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统MUST为GORM相关的4条增强规则提供结构化检测算法（字段使用追踪、批量操作模式识别、WHERE条件解析、字段顺序验证）
- **FR-002**: 系统MUST为命名日志相关的5条增强规则提供语义分析能力（调用层次构建、日志语句追踪、通用命名识别、提前返回模式检测）
- **FR-003**: 系统MUST为代码组织相关的4条增强规则提供结构分析能力（函数LOC计算、参数位置解析、import分组验证、返回值类型分析）
- **FR-004**: 系统MUST创建7个全面的错误代码测试文件，覆盖所有97条规则的违规示例（按agent和优先级组织）
- **FR-005**: 系统MUST创建1个全面的优秀代码示例文件，展示所有97条规则的正确实现（用于假阳性率验证）
- **FR-006**: 系统MUST提供增强的验证配置文件（expected_issues_v2.json），记录所有97条规则的预期检出数量和规则ID映射
- **FR-007**: 系统MUST提供增强的验证脚本（validate_results_v2.sh），计算每条规则检出率、分优先级统计、综合得分和假阳性率
- **FR-008**: 系统MUST为每个agent的SKILL.md添加"检测算法与置信度"章节，记录多轮检测策略（模式→结构→语义）和每条规则的置信度水平
- **FR-009**: 系统MUST在orchestrator/SKILL.md中添加"检测能力"章节，记录按优先级的整体检出率、已知限制和综合检测方法论
- **FR-010**: 系统MUST保持现有的4-agent并行架构不变，仅增强各agent内部的检测深度（避免架构重构风险）
- **FR-011**: 系统MUST保持所有审查结果输出为中文格式（问题描述、修改建议均为中文，代码和文件路径保持原样）
- **FR-012**: 系统MUST保留现有测试文件（user_service_bad.go、user_service_good.go）用于回归测试，新测试文件作为补充

### Key Entities *(include if feature involves data)*

- **Detection Rule**: 代表FUTU_GO_STANDARDS.md中的单条审查规则，包含规则ID（如1.3.7）、规则名称、优先级（P0/P1/P2）、检测算法类型（模式/结构/语义）、置信度水平（65%-95%）、已知限制描述。
- **Test Case File**: 代表包含特定规则违规示例的Go源代码文件，按agent分类（gorm/error-safety/naming-logging/organization）和优先级（P0/P1/P2）组织，包含预期问题数量和规则ID列表。
- **Validation Report**: 代表测试运行后生成的检出率报告，包含每条规则检出情况（检出/未检出）、分优先级统计（P0/P1/P2检出率）、综合得分（加权平均）、假阳性率、详细问题列表。
- **Agent Enhancement**: 代表单个agent的检测能力增强配置，包含增强的规则列表、新增的检测算法描述、多轮检测策略、置信度矩阵、已知限制说明。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: P0级别规则检出率达到95%以上（至少检出22/23条规则的违规实例）
- **SC-002**: P1级别规则检出率达到92%以上（至少检出40/43条规则的违规实例）
- **SC-003**: P2级别规则检出率达到88%以上（至少检出27/31条规则的违规实例）
- **SC-004**: 综合检出率（加权平均：P0×0.5 + P1×0.3 + P2×0.2）达到93%以上
- **SC-005**: 测试用例覆盖率达到100%（97/97条规则都有对应的测试用例）
- **SC-006**: 假阳性率（误报率）控制在2%以内（在优秀代码示例上产生的错误警告数量≤2个）
- **SC-007**: 单次代码审查完成时间不超过当前时间的2倍（对于200行代码文件，审查时间≤当前的2倍）
- **SC-008**: 所有97条规则都有明确记录的置信度水平（65%-95%范围）和检测算法说明
- **SC-009**: 所有agent都有完整的"检测算法与置信度"文档章节，包含规则置信度矩阵和已知限制描述
- **SC-010**: 能够通过自动化验证脚本在5分钟内完成全套测试并生成详细的检出率报告

## Assumptions

- **Assumption 1**: LLM（Claude Sonnet 4.5）有足够的代码理解能力来进行结构分析和语义分析（如字段使用追踪、调用层次构建），无需外部AST工具支持
- **Assumption 2**: 当前4-agent并行架构足够支持增强的检测能力，无需引入新的agent或改变agent间的协调方式
- **Assumption 3**: 测试文件可以通过人工编写包含已知违规模式的Go代码来创建，无需从实际生产代码中提取
- **Assumption 4**: validation脚本可以通过解析代码审查输出文本（中文格式）来提取规则ID和问题计数，无需结构化API
- **Assumption 5**: 95%的目标检出率是指"在测试用例中能检出的比例"，而非"在任意真实代码中的理论最大检出率"（某些规则如动态字段访问可能永远无法100%检出）
- **Assumption 6**: 置信度水平的评估基于规则的检测难度和已知限制，而非通过大量真实代码的统计验证
- **Assumption 7**: 2%的假阳性率目标是基于优秀代码示例测试，实际生产代码中的误报率可能略高但仍在可接受范围
- **Assumption 8**: 审查时间增加不超过2倍是合理的性能权衡，用户愿意接受稍长的审查时间以换取更高的检出率
- **Assumption 9**: 综合得分的加权公式（P0×0.5 + P1×0.3 + P2×0.2）能够准确反映不同优先级规则的重要性差异
- **Assumption 10**: 将测试文件按agent和优先级分类组织（7个bad文件+1个good文件）比单个巨大的测试文件更易于维护和调试
