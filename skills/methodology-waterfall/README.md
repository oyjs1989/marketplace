**版本**: 1.0.0

输出: 完整的SDD文档模板(中文)
```

## 核心能力

### 1. 6阶段生命周期
```
需求分析 → 系统设计 → 实施/编码 → 测试 → 部署 → 维护
    ↓         ↓          ↓        ↓      ↓      ↓
  需求文档   设计文档   源代码   测试报告 部署文档 维护日志
    ↓         ↓          ↓        ↓      ↓
  ✓审批    ✓审批      ✓审批    ✓审批  ✓验收
```

### 2. 文档模板
- **需求规格说明书 (SRS)** - IEEE 830标准
- **系统设计文档 (SDD)** - 架构/详细设计
- **测试计划和测试用例**
- **部署计划**
- **用户手册**

### 3. 阶段门控
每个阶段结束有严格的质量门控:
- 交付物完整性检查
- 文档评审和签字
- 通过标准验证
- 变更控制

### 4. 变更管理
严格的变更控制流程:
- 变更请求 (CR)
- 影响分析
- 变更评审委员会 (CCB)
- 变更实施和验证

## 瀑布模型阶段

| 阶段 | 交付物 | 门控标准 | 典型周期 |
|------|--------|---------|---------|
| 需求分析 | SRS文档 | 需求评审通过 | 2-4周 |
| 系统设计 | SDD文档 | 设计评审通过 | 3-6周 |
| 实施/编码 | 源代码+单元测试 | 代码审查通过 | 8-12周 |
| 测试 | 测试报告 | 测试完成标准 | 4-6周 |
| 部署 | 部署文档 | 验收签字 | 1-2周 |
| 维护 | 维护日志 | 持续 | - |

## 输出格式

标准化的中文瀑布式项目文档:

1. **项目计划** - 阶段划分/里程碑/甘特图
2. **需求规格说明书 (SRS)** - 功能/非功能需求
3. **系统设计文档 (SDD)** - 架构/详细设计
4. **测试计划** - 测试策略/用例/报告
5. **部署计划** - 部署步骤/回滚方案
6. **变更管理** - CR流程/影响分析

## 适用场景

### ✅ 适合使用
- 需求完全确定且不会变化
- 项目规模大,复杂度高
- 严格监管行业 (金融、医疗、航空)
- 需要详细文档和可追溯性
- 外包项目或固定价格合同
- 团队地理分布,需要明确分工

### ❌ 不适合使用
- 需求不确定或频繁变化
- 需要快速反馈和迭代
- 创新性或探索性项目
- 创业公司和快速变化的市场
- 用户需要高度参与

## 优缺点

### 优点 ✅
- 结构清晰,易于理解和管理
- 阶段明确,便于计划和跟踪
- 文档完整,可追溯性强
- 适合需求稳定的大型项目
- 合同固定,风险可控
- 便于人员分工和外包

### 缺点 ❌
- 灵活性差,难以应对需求变化
- 反馈周期长,问题发现晚
- 前期投入大,风险集中在后期
- 用户参与少,可能偏离实际需求
- 文档负担重
- 不适合创新性项目

## 改进变体

### 迭代式瀑布
将瀑布阶段缩短并迭代,增加反馈机会

### V模型
强调测试与开发阶段的对应,质量保证更好:
```
需求分析 ←→ 验收测试
系统设计 ←→ 系统测试
详细设计 ←→ 集成测试
编码    ←→ 单元测试
```

## 与其他技能集成

problem-solving → 系统化分析需求
  ↓
methodology-waterfall → 编写规范SRS
  ↓
完整需求文档
```

methodology-waterfall → 初步设计方案
  ↓

  ↓

```
methodology-waterfall → 收到变更请求
  ↓
decision-support → 多准则评估变更
  ↓
CCB决策
```

## 常见挑战

❌ **需求变更频繁** → ✅ 严格变更控制,评估影响
❌ **文档负担重** → ✅ 模板化,工具辅助
❌ **反馈周期长** → ✅ 迭代式瀑布,增加检查点
❌ **用户参与少** → ✅ 定期评审会,原型演示
