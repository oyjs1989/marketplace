# 问题解决框架技能测试指南

本文档提供7个问题解决框架技能的测试用例和验证方法。

## 测试概述

### 测试范围

本测试覆盖以下7个技能:
1. problem-solving (主编排器 + 5个代理)
2. decision-support (决策支持)
3. risk-assessment (风险评估)
4. cost-benefit-analysis (成本效益分析)
5. methodology-agile (敏捷方法论)
6. methodology-devops (DevOps方法论)
7. methodology-waterfall (瀑布方法论)

### 测试目标

- ✅ 验证每个技能可独立调用
- ✅ 验证输出格式为中文
- ✅ 验证输出内容完整性
- ✅ 验证技能互操作性
- ✅ 验证集成场景

## 单元测试

### 测试1: problem-solving 主编排器

**测试输入**:
```
帮我分析一个性能问题: 用户登录接口响应时间达到2秒,主要瓶颈在数据库查询和Session创建环节。
```

**验收标准**:
- [ ] 5个代理的分析结果都存在
- [ ] 输出全部为中文
- [ ] 包含以下分析视角:
  - 系统化思维: 系统边界、关键依赖、杠杆点
  - 建模与抽象: 核心概念、抽象模型
  - 分解策略: 层次结构、子问题
  - 迭代方法: 迭代周期、反馈机制
  - 模式识别: 识别的模式、适用方案
- [ ] 有综合解决方案
- [ ] 有下一步行动清单

### 测试2: decision-support

**测试输入**:
```
帮我选择API网关方案: Kong vs Nginx vs Traefik。评估标准: 成本(30%)、性能(25%)、易用性(25%)、生态系统(20%)。
```

**验收标准**:
- [ ] 生成决策矩阵
- [ ] 包含加权评分
- [ ] 有推荐方案及理由
- [ ] 有敏感度分析
- [ ] 输出为中文

### 测试3: risk-assessment

**测试输入**:
```
评估微服务架构迁移的风险
```

**验收标准**:
- [ ] 风险清单 (技术/项目/业务类别)
- [ ] 风险矩阵 (概率×影响)
- [ ] 优先级排序 (P0/P1/P2)
- [ ] 缓解策略
- [ ] 输出为中文

### 测试4: cost-benefit-analysis

**测试输入**:
```
评估云迁移项目的成本效益
```

**验收标准**:
- [ ] 成本识别完整 (一次性/持续/隐性)
- [ ] 效益量化有依据
- [ ] 计算了5个经济指标 (ROI/回收期/NPV/IRR/成本效益比)
- [ ] 包含情景分析 (最佳/最可能/最差)
- [ ] 有敏感度分析
- [ ] 输出为中文

### 测试5: methodology-agile

**测试输入**:
```
帮我规划下个Sprint: 开发用户认证功能
```

**验收标准**:
- [ ] Sprint信息设置
- [ ] Sprint目标定义
- [ ] 用户故事符合INVEST原则
- [ ] 任务分解
- [ ] 风险和依赖识别
- [ ] 生成Sprint Backlog
- [ ] 输出为中文

### 测试6: methodology-devops

**测试输入**:
```
为Node.js微服务项目设计CI/CD管道
```

**验收标准**:
- [ ] 管道阶段清晰 (构建→测试→部署)
- [ ] 工具链推荐
- [ ] 部署策略选择 (蓝绿/金丝雀/滚动)
- [ ] 监控指标定义
- [ ] 告警策略
- [ ] 输出为中文

### 测试7: methodology-waterfall

**测试输入**:
```
为银行核心系统编写需求规格说明书
```

**验收标准**:
- [ ] 符合IEEE 830标准结构
- [ ] 功能需求详细 (输入-处理-输出)
- [ ] 非功能需求量化
- [ ] 包含用例图/流程图占位符
- [ ] 可追溯性矩阵
- [ ] 输出为中文

## 集成测试

### 集成场景1: 微服务架构设计完整流程

**目标**: 验证多个技能协同工作

**步骤**:
```
1. 使用 problem-solving 分析微服务架构设计
   输入: "帮我设计一个电商平台的微服务架构"

2. 使用 risk-assessment 评估风险
   输入: "评估上述微服务架构的风险"

3. 使用 decision-support 比较方案
   输入: "比较服务网格方案: Istio vs Linkerd vs Consul Connect"

4. 使用 methodology-agile 进行Sprint规划
   输入: "为微服务架构实施制定Sprint计划"
```

**验收标准**:
- [ ] 4个技能顺序调用成功
- [ ] 后续技能能引用前面技能的输出
- [ ] decision-support正确集成risk-assessment结果
- [ ] 全流程输出中文一致
- [ ] 最终交付完整的设计方案+风险报告+实施计划

### 集成场景2: 技术选型决策

**目标**: 验证决策支持与风险、成本效益分析的集成

**步骤**:
```
1. 使用 decision-support 识别备选方案
   输入: "比较数据库方案: MongoDB vs PostgreSQL vs MySQL"

2. 使用 risk-assessment 评估每个方案的风险
   输入: "评估MongoDB和PostgreSQL的技术风险"

3. 使用 cost-benefit-analysis 进行经济评估
   输入: "比较MongoDB和PostgreSQL的成本效益"

4. 使用 decision-support 综合推荐
   输入: "基于上述风险和成本分析,推荐最佳数据库方案"
```

**验收标准**:
- [ ] decision-support能整合risk-assessment和cost-benefit-analysis结果
- [ ] 最终决策矩阵包含风险因素和经济因素
- [ ] 推荐方案有充分依据
- [ ] 全流程输出中文

### 集成场景3: 方法论选择

**目标**: 验证3个方法论技能的独立性和适用性

**步骤**:
```
1. 使用 methodology-agile 评估适用性
   输入: "这个项目适合用敏捷方法吗: 需求不确定,需要快速反馈"

2. 使用 methodology-devops 评估适用性
   输入: "这个项目需要DevOps实践吗: 频繁发布,微服务架构"

3. 使用 methodology-waterfall 评估适用性
   输入: "这个项目适合瀑布模型吗: 银行核心系统,需求稳定"
```

**验收标准**:
- [ ] 每个方法论正确评估适用性
- [ ] 提供清晰的适用/不适用理由
- [ ] 输出为中文

## 输出质量检查

### 中文输出验证

所有技能输出必须符合:
- [ ] 问题描述: 中文
- [ ] 分析过程: 中文
- [ ] 建议和结论: 中文
- [ ] 技术术语: 首次使用英文+中文,后续使用中文
- [ ] 代码片段: 保持原语言,注释中文
- [ ] 文件路径: 保持原格式

### 内容完整性检查

每个技能输出应包含:
- [ ] 标题和章节结构清晰
- [ ] 有执行摘要/概述
- [ ] 有详细分析过程
- [ ] 有结论和建议
- [ ] 有下一步行动 (如适用)
- [ ] 有检查清单 (如适用)

### 格式规范检查

- [ ] 使用Markdown格式
- [ ] 标题层次正确 (# ## ###)
- [ ] 列表格式统一
- [ ] 表格对齐
- [ ] 代码块使用正确的语言标识

### 并发测试

验证多个技能同时调用不会冲突或互相干扰。

## 回归测试

每次技能更新后,重新执行以下测试:
1. 所有单元测试
2. 集成场景1和2
3. 输出质量检查
