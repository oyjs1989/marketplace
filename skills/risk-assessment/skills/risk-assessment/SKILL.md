# 风险评估技能

系统化的风险评估技能,帮助识别、分析、量化和管理软件开发与IT系统中的各类风险。基于COSO ERM和ISO 31000框架。

## 何时使用此技能

当用户请求以下任何内容时使用此技能:
- "评估风险"
- "风险分析"
- "识别风险"
- "风险管理"
- "可能出什么问题"
- "失败模式"
- "风险矩阵"
- "风险评估"
- "潜在问题"

## 核心能力

### 1. 风险识别 (Risk Identification)
系统化识别项目或系统中的各类风险:
- **技术风险**: 架构、技术选型、技术债务、性能
- **项目风险**: 进度、资源、依赖、沟通
- **业务风险**: 市场、竞争、合规、战略对齐
- **安全风险**: 数据泄露、攻击、漏洞、合规
- **运维风险**: 可用性、可维护性、灾难恢复

### 2. 风险分析 (Risk Analysis)
深入分析每个风险的特征:
- **根因分析**: 识别风险的根本原因
- **影响分析**: 评估风险对项目/系统的影响
- **触发条件**: 识别风险发生的前置条件
- **连锁反应**: 分析风险的级联效应

### 3. 风险量化 (Risk Quantification)
使用定性和定量方法量化风险:
- **概率评估**: 高/中/低 或 百分比
- **影响评估**: 严重/中等/轻微 或 损失金额
- **风险值计算**: 风险值 = 概率 × 影响
- **风险矩阵**: 可视化风险分布

### 4. 风险优先级排序
基于风险值确定处理优先级:
- **P0 (关键风险)**: 高概率 + 高影响
- **P1 (重要风险)**: 中高概率 + 中高影响
- **P2 (一般风险)**: 低概率或低影响

### 5. 风险应对策略
为每个风险制定缓解策略:
- **规避 (Avoid)**: 改变计划以完全避免风险
- **降低 (Mitigate)**: 采取措施降低概率或影响
- **转移 (Transfer)**: 将风险转移给第三方
- **接受 (Accept)**: 接受风险及其后果

## 工作流程

### 第1步: 风险识别
使用多种方法识别潜在风险:

#### 识别方法

**1. 头脑风暴 (Brainstorming)**
```markdown
- [ ] 召集跨职能团队
- [ ] 从不同视角思考风险
- [ ] 鼓励开放讨论
- [ ] 记录所有潜在风险
```

**2. 检查清单 (Checklist)**
```markdown
技术风险检查:
- [ ] 技术选型是否成熟?
- [ ] 技术团队是否有经验?
- [ ] 是否存在技术债务?
- [ ] 性能是否满足要求?
- [ ] 可扩展性是否足够?

项目风险检查:
- [ ] 需求是否清晰?
- [ ] 资源是否充足?
- [ ] 时间是否现实?
- [ ] 依赖是否可控?
- [ ] 团队是否稳定?

业务风险检查:
- [ ] 市场需求是否存在?
- [ ] 竞争态势如何?
- [ ] 是否符合法规?
- [ ] 战略对齐如何?
```

**3. 历史数据分析**
```markdown
- [ ] 回顾类似项目的问题
- [ ] 分析历史故障和事故
- [ ] 识别重复出现的模式
- [ ] 学习经验教训
```

**4. 专家访谈**
```markdown
- [ ] 咨询领域专家
- [ ] 访谈有经验的团队成员
- [ ] 收集行业最佳实践
- [ ] 识别隐藏风险
```

**5. SWOT分析**
```markdown
- Strengths (优势) → 可能带来的风险
- Weaknesses (劣势) → 直接的风险来源
- Opportunities (机会) → 机会丧失的风险
- Threats (威胁) → 外部风险
```

### 第2步: 风险分析
对每个识别的风险进行深入分析:

```markdown
风险编号: R-001
风险名称: [风险简短描述]

根因分析:
- 直接原因: [...]
- 根本原因: [...]
- 相关因素: [...]

影响分析:
- 对进度的影响: [...]
- 对成本的影响: [...]
- 对质量的影响: [...]
- 对范围的影响: [...]

触发条件:
- 前置条件: [...]
- 预警信号: [...]

连锁反应:
- 次级风险1: [...]
- 次级风险2: [...]
```

### 第3步: 风险量化

#### 概率评估
使用定性或定量方法:

**定性评估**:
| 级别 | 描述 | 发生可能性 |
|------|------|-----------|
| 高 | 很可能发生 | >60% |
| 中 | 可能发生 | 20-60% |
| 低 | 不太可能发生 | <20% |

**定量评估**:
- 基于历史数据
- 专家估算
- 蒙特卡洛模拟

#### 影响评估
评估风险发生时的影响:

**定性评估**:
| 级别 | 进度影响 | 成本影响 | 质量影响 |
|------|---------|---------|---------|
| 严重 | >1个月延迟 | >30%超支 | 核心功能不可用 |
| 中等 | 1-4周延迟 | 10-30%超支 | 重要功能受影响 |
| 轻微 | <1周延迟 | <10%超支 | 次要功能受影响 |

**定量评估**:
- 延迟天数
- 成本金额
- 缺陷数量

#### 风险值计算
```
风险值 = 概率 × 影响

示例:
风险: 数据库性能不足
概率: 60% (高)
影响: 2个月延迟 (严重)
风险值: 0.6 × 9 = 5.4 (高风险)
```

### 第4步: 风险矩阵
可视化所有风险的分布:

```
影响
  ↑
严重│  [R3]     [R1][R5]   [R2]
    │
中等│           [R4]        [R7]
    │
轻微│  [R6]     [R8]
    │
    └─────────────────────────→ 概率
      低        中          高

颜色编码:
🔴 高风险 (P0): 需要立即处理
🟡 中风险 (P1): 需要密切监控
🟢 低风险 (P2): 接受或延后处理
```

### 第5步: 风险优先级排序
基于风险值和业务影响确定优先级:

```markdown
P0 (关键风险) - 必须在项目开始前或早期解决:
- 高概率 + 高影响
- 影响项目成败的风险
- 无可接受的应对措施

P1 (重要风险) - 需要在项目执行中持续监控:
- 中概率 + 高影响 或 高概率 + 中影响
- 显著影响但可管理
- 有可行的缓解措施

P2 (一般风险) - 接受或根据资源情况处理:
- 低概率或低影响
- 不影响关键目标
- 成本效益比低的风险
```

### 第6步: 风险应对策略
为每个风险制定应对计划:

#### 规避策略 (Avoid)
```markdown
适用场景: 不可接受的高风险
行动: 改变计划以消除风险

示例:
风险: 新技术栈团队不熟悉
应对: 使用团队熟悉的成熟技术栈
```

#### 降低策略 (Mitigate)
```markdown
适用场景: 可以采取措施降低的风险
行动: 降低概率或减轻影响

示例:
风险: 第三方API不稳定
应对:
- 添加重试机制 (降低影响)
- 实现熔断器 (降低影响)
- 建立服务降级 (降低影响)
- 与供应商建立SLA (降低概率)
```

#### 转移策略 (Transfer)
```markdown
适用场景: 可以转移给第三方的风险
行动: 通过保险、外包或合同转移

示例:
风险: 数据中心故障
应对: 使用云服务商,风险转移给供应商
```

#### 接受策略 (Accept)
```markdown
适用场景: 低影响或应对成本过高
行动: 接受风险,准备应急计划

示例:
风险: 小概率的罕见错误
应对: 接受,建立监控和快速响应机制
```

### 第7步: 持续监控
建立风险监控和更新机制:

```markdown
- [ ] 定义风险指标和阈值
- [ ] 建立监控仪表盘
- [ ] 定期评审风险状态
- [ ] 识别新出现的风险
- [ ] 更新风险应对计划
```

## 风险类别详解

### 技术风险
- 技术选型不当
- 技术债务累积
- 性能瓶颈
- 可扩展性限制
- 技术过时
- 集成复杂性

### 项目风险
- 需求不清晰或变更频繁
- 时间表不现实
- 资源不足
- 团队技能缺口
- 依赖延迟
- 沟通不畅

### 业务风险
- 市场需求变化
- 竞争加剧
- 法规变化
- 战略不对齐
- 预算削减
- 利益相关者冲突

### 安全风险
- 数据泄露
- 未授权访问
- 漏洞利用
- 合规违规
- 供应链攻击
- 内部威胁

### 运维风险
- 系统可用性
- 性能衰退
- 运维复杂度
- 灾难恢复
- 监控盲点
- 变更管理
