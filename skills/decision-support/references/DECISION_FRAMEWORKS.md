## 决策分析概述

决策支持系统框架涉及风险分析和优化过程,通过系统化的、分阶段的决策分析框架,将成本、效益和风险分析整合在一起,形成闭环决策支持流程。

## 方法1: 加权评分法 (Weighted Scoring Method)

### 概述
最常用和直观的多准则决策方法,通过为每个标准分配权重,对方案评分并计算加权总分。

### 何时使用
- ✅ 有3-7个备选方案
- ✅ 有5-10个评估标准
- ✅ 标准可量化或定性评估
- ✅ 团队对权重有共识
- ✅ 时间有限,需要快速决策

### 操作步骤

#### 步骤1: 定义评估标准
```markdown
- [ ] 头脑风暴所有可能的标准
- [ ] 分类(技术/业务/经济/风险)
- [ ] 筛选出5-10个关键标准
- [ ] 为每个标准编写清晰定义
- [ ] 确保标准相互独立
```

**标准选择原则**:
- SMART原则 (Specific, Measurable, Achievable, Relevant, Time-bound)
- 覆盖关键成功因素
- 避免重复或重叠
- 可客观评估

#### 步骤2: 确定权重
```markdown
- [ ] 识别最重要的标准
- [ ] 使用百分比分配权重(总和100%)
- [ ] 或使用两两比较确定相对重要性
- [ ] 验证权重反映了决策优先级
- [ ] 获得利益相关者同意
```

**权重分配方法**:

**方法A: 直接分配**
```
最重要标准: 20-30%
重要标准: 10-20%
次要标准: 5-10%
辅助标准: <5%

总和必须 = 100%
```

**方法B: 相对重要性矩阵**
```
        标准1  标准2  标准3
标准1    1     3     5
标准2   1/3    1     3
标准3   1/5   1/3    1

计算每行几何平均值
归一化为权重
```

#### 步骤3: 评分
```markdown
- [ ] 定义评分标准(1-5分或1-10分)
- [ ] 对每个方案在每个标准上评分
- [ ] 记录评分依据
- [ ] 多人独立评分后取平均
- [ ] 处理分歧和争议
```

**评分尺度建议**:

**5分制**:
- 1分: 非常差
- 2分: 差
- 3分: 一般
- 4分: 好
- 5分: 非常好

**10分制**:
- 1-2分: 不可接受
- 3-4分: 差
- 5-6分: 一般
- 7-8分: 好
- 9-10分: 优秀

#### 步骤4: 计算加权总分
```markdown
- [ ] 对每个方案,计算: 加权得分 = 评分 × 权重
- [ ] 汇总所有加权得分
- [ ] 得到总分
- [ ] 排序所有方案
```

**计算公式**:
```
总分 = Σ (评分i × 权重i)

其中 i = 1 到 n (n为标准数量)
```

#### 步骤5: 分析和验证
```markdown
- [ ] 识别最高得分方案
- [ ] 分析得分差异
- [ ] 识别每个方案的强项和弱项
- [ ] 进行敏感度分析
- [ ] 验证结果的合理性
```

### 实践示例

**决策**: 选择API网关方案

**备选方案**:
- 方案A: Kong
- 方案B: Nginx + Lua
- 方案C: AWS API Gateway

**评估标准和权重**:
| 标准 | 权重 |
|------|------|
| 性能 | 25% |
| 功能完整性 | 20% |
| 易用性 | 15% |
| 成本 | 20% |
| 社区支持 | 10% |
| 云原生 | 10% |

**评分矩阵** (10分制):
| 方案 | 性能 | 功能 | 易用性 | 成本 | 社区 | 云原生 | 总分 |
|------|------|------|--------|------|------|--------|------|
| Kong | 8(2.0) | 9(1.8) | 7(1.05) | 6(1.2) | 9(0.9) | 8(0.8) | **7.75** |
| Nginx | 9(2.25) | 7(1.4) | 5(0.75) | 9(1.8) | 8(0.8) | 6(0.6) | **7.6** |
| AWS AG | 7(1.75) | 8(1.6) | 9(1.35) | 5(1.0) | 7(0.7) | 10(1.0) | **7.4** |

**结论**: Kong以7.75分排名第一,优势在于功能完整性和社区支持。

## 方法2: AHP (层次分析法)

### 概述
系统化的多层次决策方法,通过两两比较确定标准权重,处理复杂的层次化决策问题。

### 何时使用
- ✅ 决策涉及多个层次
- ✅ 标准权重难以直接确定
- ✅ 需要更科学的权重确定方法
- ✅ 有时间进行详细分析
- ✅ 决策重要性高

### 操作步骤

#### 步骤1: 建立层次结构
```
目标层: 选择最佳API网关
  ↓
准则层: 性能、功能、成本、易用性
  ↓
子准则层: (如有)响应时间、吞吐量、...
  ↓
方案层: Kong、Nginx、AWS
```

#### 步骤2: 构造判断矩阵
对准则层的标准进行两两比较:

**Saaty 1-9标度**:
| 标度 | 含义 |
|------|------|
| 1 | 同等重要 |
| 3 | 稍微重要 |
| 5 | 明显重要 |
| 7 | 强烈重要 |
| 9 | 极端重要 |
| 2,4,6,8 | 上述相邻判断的中间值 |

**示例判断矩阵**:
```
        性能  功能  成本  易用性
性能     1    3    2     5
功能    1/3   1    1/2   3
成本    1/2   2    1     3
易用性  1/5  1/3  1/3    1
```

#### 步骤3: 计算权重
```markdown
- [ ] 计算每行几何平均值
- [ ] 归一化得到权重向量
- [ ] 进行一致性检验(CI和CR)
- [ ] 如果CR>0.1,需要调整判断矩阵
```

**一致性检验**:
```
一致性比率 CR = CI / RI

CI = (λmax - n) / (n - 1)
其中:
- λmax: 最大特征值
- n: 矩阵阶数
- RI: 随机一致性指标(查表)

CR < 0.1 表示一致性可接受
```

#### 步骤4: 方案评分
对每个方案在每个标准上进行两两比较,计算得分。

#### 步骤5: 综合排序
```
总得分 = Σ (标准权重i × 方案在标准i上的得分)
```

### AHP检查清单
- [ ] 层次结构是否合理?
- [ ] 判断矩阵是否一致?(CR<0.1)
- [ ] 权重是否反映真实偏好?
- [ ] 结果是否稳健?(敏感度分析)

## 方法3: TOPSIS (逼近理想解法)

### 概述
通过计算各方案到理想解和负理想解的距离,选择最接近理想解的方案。

### 何时使用
- ✅ 有大量备选方案(>5个)
- ✅ 标准可量化
- ✅ 需要客观排序
- ✅ 避免主观判断

### 操作步骤

#### 步骤1: 构建决策矩阵
```
        标准1  标准2  标准3
方案A    x11    x12    x13
方案B    x21    x22    x23
方案C    x31    x32    x33
```

#### 步骤2: 规范化决策矩阵
```
归一化: rij = xij / √(Σ xij²)
```

#### 步骤3: 加权规范化
```
加权值: vij = rij × wj
其中 wj 是标准j的权重
```

#### 步骤4: 确定理想解和负理想解
```
理想解 V+: 每个标准的最大值(效益型)或最小值(成本型)
负理想解 V-: 每个标准的最小值(效益型)或最大值(成本型)
```

#### 步骤5: 计算距离
```
到理想解的距离: Si+ = √(Σ (vij - vj+)²)
到负理想解的距离: Si- = √(Σ (vij - vj-)²)
```

#### 步骤6: 计算相对贴近度
```
贴近度: Ci = Si- / (Si+ + Si-)

Ci ∈ [0, 1]
Ci 越大,方案越优
```

### 实践示例

**决策**: 选择云服务商

**方案**: AWS, Azure, GCP, Alibaba Cloud

**标准**: 性能(30%), 价格(25%), 功能(25%), 支持(20%)

[省略具体计算步骤]

**结果**:
- AWS: C1 = 0.72
- Azure: C2 = 0.68
- GCP: C3 = 0.65
- Alibaba: C4 = 0.58

**结论**: AWS最优

## 方法4: 决策树分析 (Decision Tree Analysis)

### 概述
适用于顺序决策和不确定性场景,通过树形结构可视化决策路径和结果。

### 何时使用
- ✅ 决策涉及多个阶段
- ✅ 存在不确定性和概率
- ✅ 需要可视化决策路径
- ✅ 需要考虑后续决策

### 操作步骤

#### 步骤1: 绘制决策树
```
决策节点 (方框)
  ├─ 方案A → 机会节点 (圆圈)
  │           ├─ 结果1 (概率p1) → 收益1
  │           └─ 结果2 (概率p2) → 收益2
  └─ 方案B → 机会节点
              ├─ 结果1 (概率p1) → 收益1
              └─ 结果2 (概率p2) → 收益2
```

#### 步骤2: 确定概率和收益
```markdown
- [ ] 为每个不确定事件估算概率
- [ ] 为每个最终结果估算收益(或成本)
- [ ] 验证概率总和 = 1
```

#### 步骤3: 计算期望值
```
期望值 = Σ (概率i × 收益i)
```

#### 步骤4: 回溯分析
从树的末端向起点计算每个节点的期望值,选择期望值最高的路径。

### 实践示例

**决策**: 是否自研还是采购CRM系统

```
决策
├─ 自研
│  ├─ 成功 (60%) → 收益: ¥500万
│  └─ 失败 (40%) → 损失: -¥200万
└─ 采购
   ├─ 适配顺利 (80%) → 收益: ¥300万
   └─ 适配困难 (20%) → 收益: ¥100万

期望值(自研) = 0.6×500 + 0.4×(-200) = 220万
期望值(采购) = 0.8×300 + 0.2×100 = 260万

结论: 采购期望值更高,风险更低
```

## 敏感度分析

### 目的
评估决策的稳健性,识别关键影响因素。

### 分析类型

#### 1. 权重敏感度
```markdown
- [ ] 改变关键标准的权重(±5%, ±10%)
- [ ] 重新计算排序
- [ ] 观察排序是否改变
- [ ] 识别对哪些权重最敏感
```

**示例**:
```
基准: 性能25%, 成本20%, 功能20%
场景1: 性能35%, 成本15%, 功能20%
  → 排序变化: A>B>C 变为 B>A>C
结论: 决策对性能权重敏感
```

#### 2. 评分敏感度
```markdown
- [ ] 改变关键方案的评分(��1分, ±2分)
- [ ] 重新计算排序
- [ ] 评估多少评分误差会改变决策
```

#### 3. 情景分析
```markdown
- [ ] 定义最佳情景 (乐观假设)
- [ ] 定义最差情景 (悲观假设)
- [ ] 定义最可能情景 (现实假设)
- [ ] 在每个情景下评估方案
```

**示例**:
```
最佳情景: 项目顺利,成本控制好
最差情景: 遇到重大技术障碍,成本超支
最可能情景: 正常进度,小幅超支

评估每个方案在3种情景下的表现
```

### 敏感度分析报告模板

```markdown
## 敏感度分析

### 关键影响因素
1. **[因素1]**: 对决策影响最大
   - 变化范围: [X%到Y%]
   - 影响: 可能改变最优方案

2. **[因素2]**: 次要影响
   - 变化范围: [X%到Y%]
   - 影响: 影响得分但不改变排序

### 决策稳健性
- **稳健**: 在合理变化范围内,最优方案不变
- **中等**: 某些变化可能改变排序
- **脆弱**: 小幅变化即可能改变决策

### 风险缓解
如果决策脆弱,考虑:
- 收集更多数据减少不确定性
- 设计灵活方案以应对变化
- 建立监控机制及时调整
```

## 决策陷阱和对策

### 常见陷阱

#### 1. 锚定偏差 (Anchoring Bias)
**特征**: 过度依赖第一个信息
**对策**:
- 独立评估每个方案
- 多人独立评分后汇总
- 延迟讨论第一印象

#### 2. 确认偏差 (Confirmation Bias)
**特征**: 只寻找支持预设结论的证据
**对策**:
- 主动寻找反对证据
- 设置"红队"挑战方案
- 盲测评估

#### 3. 沉没成本谬误 (Sunk Cost Fallacy)
**特征**: 因已投入而继续错误方向
**对策**:
- 只考虑未来成本和效益
- 忽略过去投入
- 定期重新评估

#### 4. 过度自信 (Overconfidence)
**特征**: 高估成功概率
**对策**:
- 参考历史数据
- 咨询专家意见
- 进行最差情景分析

#### 5. 群体思维 (Groupthink)
**特征**: 团队一致性压力
**对策**:
- 鼓励不同意见
- 匿名投票
- 外部评审
