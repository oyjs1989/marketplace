本文档提供敏捷方法论的详细实践指南、模板和检查清单。

> **来源**: 基于 doc.md 第3.2节 - 敏捷开发的问题解决逻辑

## 敏捷宣言核心

### 4个价值观
1. **个体和互动** > 流程和工具
2. **可工作的软件** > 详尽的文档
3. **客户协作** > 合同谈判
4. **响应变化** > 遵循计划

### 12条原则

前6条(客户满意和交付):
1. 尽早持续交付有价值软件
2. 欢迎需求变更
3. 频繁交付 (几周到几个月)
4. 业务和开发每天协作
5. 信任有动力的个人
6. 面对面沟通最有效

后6条(团队和质量):
7. 可工作软件是进度度量
8. 可持续开发速度
9. 技术卓越和良好设计
10. 简洁——最大化未完成工作
11. 自组织团队
12. 定期反思和调整

## Scrum框架实施指南

### Sprint规划清单

#### 准备阶段
```markdown
Product Owner:
- [ ] Product Backlog已refinement
- [ ] Top stories有验收标准
- [ ] 优先级清晰
- [ ] 可回答团队问题

Scrum Master:
- [ ] 预订会议室
- [ ] 准备上个Sprint数据
- [ ] 准备velocity图表
- [ ] 任务板ready

Team:
- [ ] 了解上个Sprint成果
- [ ] 预先阅读top stories
- [ ] 准备提问
```

#### 规划会议
```markdown
第一小时: What
1. [ ] PO呈现Sprint目标
2. [ ] PO介绍top stories
3. [ ] 团队提问澄清
4. [ ] 团队评估capacity
5. [ ] 团队选择stories
6. [ ] 确认Sprint目标

第二小时: How
1. [ ] 团队分解stories为任务
2. [ ] 估算任务时间
3. [ ] 识别依赖
4. [ ] 识别风险
5. [ ] 成员认领任务
6. [ ] 确认可完成性

输出检查:
- [ ] Sprint目标一句话概括
- [ ] Sprint Backlog清晰
- [ ] 任务分解完整
- [ ] 团队承诺目标
```

### 每日站会指南

#### 站会格式

**标准格式 (每人回答3个问题)**:
```markdown
1. 昨天完成: [具体成果]
2. 今天计划: [具体任务]
3. 遇到阻碍: [无 / 具体描述]
```

**Walk the Board (走任务板)**:
```markdown
1. 从右到左review任务板
2. 讨论接近完成的任务
3. 讨论阻塞的任务
4. 计划今天的工作
```

#### 站会反模式

❌ **状态报告会**:
- 问题: 对SM或PO汇报
- 解决: 团队面向团队,互相同步

❌ **问题解决会**:
- 问题: 深入讨论技术细节
- 解决: 记录问题,会后单独讨论

❌ **超时会议**:
- 问题: 经常超过15分钟
- 解决: SM严格time-box,打断偏题

❌ **沉默的站会**:
- 问题: 机械式回答,无互动
- 解决: 走任务板,问开放问题

### Sprint评审指南

#### 演示准备
```markdown
提前准备:
- [ ] 演示环境ready
- [ ] 演示数据准备好
- [ ] 演示脚本 (但不照本宣科)
- [ ] 备份计划 (如果环境问题)
- [ ] 邀请利益相关者
- [ ] 准备Product Backlog更新

演示技巧:
- ✅ 从用户视角演示
- ✅ 展示业务价值
- ✅ 讲故事而非功能列表
- ✅ 鼓励提问和反馈
- ✅ 诚实面对问题
```

#### 演示议程
```markdown
1. 开场 (5分钟)
   - Sprint目标回顾
   - 完成 vs 承诺

2. 演示 (60-75%)
   - 逐个演示完成stories
   - 鼓励试用
   - 收集反馈

3. Product Backlog讨论 (20-30%)
   - Backlog变化
   - 优先级调整
   - 下个Sprint预览

4. 总结 (5分钟)
   - 关键takeaways
   - 下一步
```

### Sprint回顾指南

#### 回顾技术选择

**1. Start-Stop-Continue**
```markdown
Start (开始做):
- 我们应该开始做什么新事情?

Stop (停止做):
- 我们应该停止做什么?

Continue (继续做):
- 我们应该继续做什么?
```

**2. Mad-Sad-Glad**
```markdown
Mad (生气): 什么让你沮丧?
Sad (伤心): 什么让你失望?
Glad (高兴): 什么让你开心?
```

**3. 4Ls**
```markdown
Liked (喜欢): 什么做得好?
Learned (学到): 学到了什么?
Lacked (缺少): 缺少什么?
Longed for (渴望): 渴望什么?
```

**4. Sailboat (帆船回顾)**
```markdown
风 (Wind): 推动我们前进的因素
锚 (Anchor): 拖慢我们的因素
礁石 (Rocks): 潜在风险
岛 (Island): 我们的目标
```

#### 回顾流程
```markdown
1. Set the Stage (5分钟)
   - 建立安全环境
   - 回顾Prime Directive
   - 选择技术

2. Gather Data (15分钟)
   - 收集事实和感受
   - 时间线/数据点
   - 每人发言

3. Generate Insights (15分钟)
   - 识别模式
   - 根因分析
   - 分组讨论

4. Decide What to Do (10分钟)
   - 投票选择改进项
   - 定义行动 (SMART)
   - 分配责任

5. Close (5分钟)
   - 感谢参与
   - 确认行动项
   - 预告下次
```

## 用户故事编写指南

### 故事模板

**标准格式**:
```markdown
作为 [用户角色]
我想要 [功能/目标]
以便 [业务价值/原因]

验收标准:
- [ ] 给定 [前置条件]
     当 [动作]
     则 [预期结果]
- [ ] 标准2
- [ ] 标准3

技术笔记 (可选):
- [实现考虑]
- [依赖项]
- [风险]
```

### INVEST检查

```markdown
I - Independent (独立)
- [ ] 可以任意顺序开发
- [ ] 不严重依赖其他故事
- [ ] 可以单独交付价值

N - Negotiable (可协商)
- [ ] 是对话的起点,非合同
- [ ] 细节可以讨论
- [ ] 实现方式灵活

V - Valuable (有价值)
- [ ] 对用户/客户有明确价值
- [ ] 可以回答"为什么"
- [ ] PO可以排优先级

E - Estimable (可估算)
- [ ] 团队可以估算大小
- [ ] 团队理解故事
- [ ] 没有太多不确定性

S - Small (小)
- [ ] 可以在一个Sprint内完成
- [ ] 建议≤13点
- [ ] 大故事需要拆分

T - Testable (可测试)
- [ ] 有明确验收标准
- [ ] 可以验证完成
- [ ] 可以自动化测试
```

### 拆分技术

**按工作流步骤拆分**:
```
大故事: 用户可以在线支付
拆分:
- 用户可以添加支付方式
- 用户可以选择支付方式
- 用户可以确认支付
- 用户可以查看支付历史
```

**按业务规则拆分**:
```
大故事: 用户可以搜索产品
拆分:
- 按名称搜索
- 按类别搜索
- 按价格范围搜索
- 高级组合搜索
```

**按数据变化拆分**:
```
大故事: 支持多种用户角色
拆分:
- 普通用户功能
- VIP用户功能
- 管理员功能
```

## 估算技术

### Planning Poker流程

```markdown
1. Product Owner读故事
2. 团队提问澄清 (2-3分钟)
3. 每人秘密选择卡片 (斐波那契)
4. 同时亮牌
5. 如果一致 → 记录估算
6. 如果不一致 → 最高和最低解释理由
7. 重新讨论和投票
8. 达成共识 (不需要完全一致,接近即可)
```

**估算参考**:
```
1点: 微不足道 (<2小时)
2点: 简单 (半天)
3点: 中等 (1天)
5点: 复杂 (2-3天)
8点: 很复杂 (一周)
13点: 考虑拆分
21点+: 必须拆分
```

### T-Shirt Sizing

快速粗略估算:
```
XS: 非常小
S: 小
M: 中
L: 大
XL: 很大

适用于:
- 早期粗略规划
- 大量backlog快速分类
- 初步容量规划
```

### Velocity计算和使用

**计算Velocity**:
```
Velocity = 过去N个Sprint完成的故事点平均值

示例:
Sprint 1: 28点
Sprint 2: 25点
Sprint 3: 30点
平均Velocity = (28+25+30)/3 = 27.7点

通常用N=3 (最近3个Sprint)
```

**使用Velocity规划**:
```
剩余工作: 150点
平均Velocity: 28点
预计需要: 150/28 = 5.4个Sprint

发布日期 = 当前日期 + 6个Sprint
```

**Velocity注意事项**:
```markdown
- ✅ 用于团队规划
- ✅ 识别趋势
- ✅ 容量预测

- ❌ 不比较团队间velocity
- ❌ 不作为团队绩效指标
- ❌ 新团队前几个Sprint不稳定
```

## Burndown Chart (燃尽图)

### Sprint Burndown

**横轴**: Sprint天数
**纵轴**: 剩余工作(点或小时)

**理想线**: 从Sprint起始工作量直线下降到0
**实际线**: 每天实际剩余工作量

**模式识别**:
```markdown
理想趋势 ↘️:
实际线稳定向下,接近理想线

早期滞后 ⏸️:
前半程平坦,后半程陡降
→ 警示: 可能无法完成承诺

Scope增加 ↗️:
实际线上升
→ 警示: Sprint中增加了工作

过度承诺 📉:
最后几天仍有大量工作
→ 警示: 估算不准或过度承诺
```

### Release Burndown

**横轴**: Sprint编号
**纵轴**: 剩余Product Backlog点数

**用途**: 预测发布日期

## 敏捷度量指标

### 关键指标

**1. Velocity (速度)**
```
定义: 每Sprint完成的故事点
用途: 容量规划,趋势分析
目标: 稳定并可预测
```

**2. Sprint Commitment vs Completion**
```
定义: 承诺点数 vs 完成点数
计算: 完成率 = 完成/承诺 × 100%
目标: >80%,趋向稳定
```

**3. Cycle Time (周期时间)**
```
定义: 故事从"开始"到"完成"的时间
用途: 识别瓶颈,改进流程
目标: 减少并稳定
```

**4. Lead Time (前置时间)**
```
定义: 从backlog添加到完成的时间
用途: 客户视角的响应时间
目标: 减少
```

**5. Defect Rate (缺陷率)**
```
定义: 每Sprint发现的bug数量
分类: P0/P1/P2
目标: 减少,特别是P0/P1
```

**6. Code Coverage (代码覆盖率)**
```
定义: 测试覆盖的代码百分比
目标: >80% (单元测试)
追踪: 趋势向上
```

**7. Team Happiness (团队满意度)**
```
定义: 每Sprint回顾的团队投票
方式: 1-5分或emoji
目标: >4分,稳定或上升
```

### 反指标 (Vanity Metrics)

❌ **不推荐的指标**:
- 代码行数 (LOC)
- 工作时长
- 个人velocity
- 团队间velocity对比
- 缺陷关闭数 (不看新增)

## 敏捷工具和技术

### 物理工具

**任务板 (物理板)**:
```
To Do | In Progress | Done

好处:
- 高可见性
- 触感 (移动卡片有满足感)
- 团队聚集点
- 灵活可定制

限制:
- 远程团队困难
- 不便于搜索和报告
```

**便利贴和卡片**:
```
- 不同颜色代表不同类型
- 大小表示重要性
- 物理操作促进协作
```

### 数字工具

**常用工具**:
```
Jira: 功能强大,适合大团队
Trello: 简单直观,适合小团队
Azure DevOps: 微软生态集成
ClickUp: 灵活可定制
Linear: 现代简洁
```

**选择标准**:
```markdown
- [ ] 易用性 (学习曲线)
- [ ] 团队规模支持
- [ ] 与现有工具集成
- [ ] 报告和度量
- [ ] 成本
- [ ] 移动支持
- [ ] 自定义能力
```

## 分布式敏捷实践

### 远程Scrum调整

**挑战和解决**:

**1. 沟通困难**
```markdown
挑战: 缺少面对面互动
解决:
- 视频优先 (不只是音频)
- 更频繁的check-in
- 异步沟通工具 (Slack)
- 虚拟咖啡 (非正式交流)
```

**2. 时区差异**
```markdown
挑战: 团队分布多个时区
解决:
- 找到重叠时间 (core hours)
- 轮流牺牲 (公平分担)
- 异步更新 (录屏,文档)
- 明确响应时间SLA
```

**3. 协作困难**
```markdown
挑战: 结对编程、白板讨论困难
解决:
- 屏幕共享 + 远程控制
- 在线白板 (Miro, Mural)
- 代码共享工具 (Live Share)
```

**4. 建立信任**
```markdown
挑战: 远程难以建立团队凝聚力
解决:
- 定期团队活动 (虚拟游戏)
- 开摄像头 (面对面感)
- 一对一关怀
- 偶尔线下聚会
```

### 远程会议最佳实践

**Sprint规划**:
```markdown
- 提前分享材料
- 使用在线任务板
- 在线投票工具 (Planning Poker)
- 休息时间 (远程更累)
```

**每日站会**:
```markdown
- 固定时间
- 视频必开
- 使用在线看板
- 限制15分钟严格执行
- 录制 (时区无法参加的人)
```

**Sprint回顾**:
```markdown
- 使用在线回顾工具
- 匿名投票
- 小组讨论 (breakout rooms)
- 实时协作文档
```

## 最佳实践总结

### 团队层面

✅ 保持团队稳定和全职
✅ 跨职能 (减少依赖)
✅ 合理规模 (5-9人)
✅ 物理或虚拟的团队空间
✅ 持续改进文化

### Product Owner层面

✅ 单一PO (避免多头指挥)
✅ 可访问和投入
✅ 清晰的产品愿景
✅ 果断的优先级决策
✅ 信任团队

### Scrum Master层面

✅ 仆人式领导
✅ 移除阻碍,不解决问题
✅ 教练而非指挥
✅ 保护团队
✅ 持续学习敏捷

### 技术层面

✅ 持续集成和部署
✅ 自动化测试
✅ 代码审查
✅ 重构时间
✅ 技术卓越追求
